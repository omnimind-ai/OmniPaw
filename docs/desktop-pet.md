# 桌宠功能包约束

## 包边界

- MUST：桌宠的 BrowserWindow 生命周期、窗口专属 IPC、HTML 入口和专属 renderer 归属 `packages/desktop-pet/`。
- MUST：桌宠通用业务规则、持久化和跨层类型分别留在 `core/` 与 `shared/`，不得复制进功能包。
- MUST：应用启动层只能装配桌宠 controller，不得重新承载桌宠窗口状态机或输入逻辑。
- MUST：根目录只保留主 renderer 入口；桌宠 HTML 入口必须留在包内。

## 窗口与输入不变量

- MUST：视觉窗口保持透明、置顶和点击穿透；可交互输入由独立命中窗口承载。
- MUST：视觉窗口与命中窗口共享同一权威位置和缩放语义，任何移动、显示、隐藏、缩放和形象布局变化都必须保持二者同步。
- MUST：命中区域来自视觉层实际可交互区域，不得用与视觉内容无关的固定全窗区域替代。
- MUST：拖动期间窗口移动、指针捕获、结束清理和异常中止保持一致，不得留下悬空拖动状态。
- MUST：文件投递和指针拖动共享输入窗口时，不得互相破坏命中、点击穿透或拖动生命周期。
- MUST：面板和气泡的定位以桌宠窗口状态为依据，并在显示器边界内保持可见。

## Renderer 边界

- MUST：桌宠 renderer 通过 `appBridge` 使用主进程能力，不得直接导入 Electron、Node、core 或文件系统 API。
- MUST：视觉状态、DOM 展示和输入控制保持独立责任；状态变化不得直接绕过 bridge 操纵其他窗口。
- MUST：所有跨窗口命令、事件和 payload 使用 `shared/` 契约。
- MUST：订阅、计时器、指针捕获、object URL 和其他外部资源在卸载或窗口关闭时释放。
- MUST：桌宠入口满足项目 CSP；首帧所需样式不得依赖被 CSP 阻止的注入方式。

## 状态与形象

- MUST：桌宠视觉状态由单一状态机决定，任务、拖动和出现/完成过渡不得由多个入口竞态修改。
- MUST：异步动画切换必须防止旧计时器覆盖新状态。
- MUST：形象包解析、回退和安全校验归属 core；renderer 只消费已经解析的资源契约。
- MUST：资源缺失或加载失败时保持可见、可恢复的回退状态，不得破坏输入窗口几何关系。

## 验证约束

- MUST：窗口、命中区、拖动或文件投递变更通过桌宠输入 smoke。
- MUST：形象包契约或状态资源变更通过形象包 smoke。
- MUST：入口、preload、主进程装配或构建配置变更通过完整构建。
- SHOULD：涉及透明度、置顶、点击穿透或多窗口对齐的改动进行真实 Electron 窗口验证；无法自动验证时必须说明。
